## 1. 🔴 Release pipeline has a latent bug

**This is the single most important finding.** `scripts/release.ts` (142 lines) implements the `publishConfig.exports` swap flow the repo documents as canonical. It is never called by CI.

### Evidence

`.github/workflows/release.yml:50`

```yaml
- uses: changesets/action@v1
  with:
    version: bun run version
    publish: bun run publish:packages # ← this
```

Root `package.json:44`

```json
"publish:packages": "changeset publish"
```

So the release path is: changesets-action → `changeset publish` → `npm pack` for each package → npm registry. The swap that `release.ts` performs — replacing src-pointing `exports` with the `publishConfig.exports` that points at `dist/*.js` — is done by **`scripts/prepack-exports.ts` / `postpack-exports.ts`** hooks wired into `packages/ui/package.json:1501-1502` and `packages/primitives/package.json:1501-1502`.

**So `release.ts` is dead code.** It is never invoked. Any time a contributor opens it to debug a publish problem, they will be reading a script that is not the one running. That is a timebomb.

### Impact

- ~142 lines of code that document a release flow that isn't the release flow.
- Drift risk: if the prepack hook and `release.ts` fall out of alignment (they are _already_ slightly different in how they resolve `workspace:*` dependencies), the wrong half will be edited during an incident.
- New contributors cannot trust the docs vs. the code.

### Fix

Pick one of two paths — do **not** keep both:

**Option A (recommended): delete `release.ts`.** The changesets-action + prepack/postpack hooks are the working path. Update `CLAUDE.md` to describe that flow. Move any manual-publish convenience (OTP handling, dry-run flags) into `scripts/publish.ts`.

**Option B: wire `release.ts` in.** Change `release.yml:50` to `publish: bun run scripts/release.ts` and delete the prepack/postpack hooks. Riskier — changesets-action expects `changeset publish` semantics for changelog commit SHAs.

Either way, the outcome is **one release script, not two** (three if you count prepack/postpack as one). See [§3](#3--release--publish-duplicate-swap-logic) for the code-level consolidation that falls out of this.

## 2. 🔴 Dead-but-shipped primitives

`CLAUDE.md:124` contains a hard rule:

> **Layout is raw CSS grid** — do not use `Grid`, `Stack`, or `Flex` components.

And `@dryui/lint` enforces `dryui/no-width` and bans flexbox. Yet:

```
packages/primitives/src/flex/flex.svelte      ← exists
packages/primitives/src/stack/stack.svelte    ← exists
packages/primitives/src/grid/grid.svelte      ← exists
packages/primitives/src/grid/grid-area.svelte ← exists
```

Verified: **nothing in the repo imports them.** `rg "primitives/(flex|stack|grid)"` across `packages/` and `apps/` returns zero results.

### Impact

- Shipped to npm under `@dryui/primitives`. External consumers can import banned layout primitives that directly contradict the library's documented philosophy.
- Adds entries to the `sync-package-exports` output that need to be kept up to date for no reason.
- Increases the bundle that `@dryui/primitives` `svelte-package` has to process on every build.

This is the project's **only** example of shipping something that violates its own stated principles. That alone makes it worth fixing.

### More dead/orphan primitives

A full directory-diff between `packages/primitives/src` and `packages/ui/src` turned up **20 primitives with no UI sibling**:

```
affix-group, app-frame, avatar-group, chat-message, empty-state, flex,
footer, graphify-out, grid, hero, logo-cloud, page-header,
selectable-tile-group, stack, stat-card, surface, use-theme-override,
user, wave-divider
```

Some of these are legitimate (utility primitives like `use-theme-override`, `affix-group`, `empty-state`). Others look like stale layout/shell components (`app-frame`, `page-header`, `hero`, `footer`, `stat-card`) that may have been superseded. **Do a triage pass.** For each, answer: is it imported by external consumers? If not, and it duplicates what a grid layout gives you for free, delete.

Conversely, `packages/ui/src` has **18 components with no primitive sibling**:

```
alert, alpha-slider, chat-thread, description-list, diagram, drop-zone,
fieldset, halftone, heading, input-group, logo-mark, option-picker,
shimmer, star-rating, text, themes, typing-indicator
```

Most are fine (pure styling: `text`, `heading`, `halftone`, `shimmer`, `themes`, etc. have no behavioral state to headless). But `option-picker`, `drop-zone`, `alpha-slider`, `star-rating`, `input-group`, `fieldset`, and `description-list` _do_ have interactive behavior — those should have primitives so external consumers can rebuild them with their own styling, consistent with the rest of the library.

### Fix

1. Delete `packages/primitives/src/{flex,stack,grid}` and re-run `bun run sync:exports`.
2. Triage the 20 primitives-only components — delete, document as "utility primitive", or add a UI sibling.
3. Triage the 7 UI-only interactive components and promote their headless behavior to primitives.

## 3. 🟠 release.ts and publish.ts duplicate swap logic

Even once §1 is resolved, `release.ts:54-90` and `publish.ts:50-88` contain **the same `swapForPublish` function** differing only in the backup filename suffix (`.release-backup` vs `.publish-backup`). `prepack-exports.ts` does the same swap a third time, backing up to `.exports-backup.json`.

### Evidence

| File                              | Lines | Purpose                                                           |
| --------------------------------- | ----- | ----------------------------------------------------------------- |
| `scripts/release.ts`              | 142   | Changeset version + batch publish with swap                       |
| `scripts/publish.ts`              | 118   | Single-package publish with swap                                  |
| `scripts/prepack-exports.ts`      | 26    | Pre-`npm pack` swap                                               |
| `scripts/postpack-exports.ts`     | 23    | Post-`npm pack` restore                                           |
| `scripts/sync-package-exports.ts` | 162   | Generate the `exports` + `publishConfig.exports` maps from source |

Every one of them touches `publishConfig.exports`. Three of them reimplement the same read-mutate-write-backup cycle.

### Fix

Extract `scripts/lib/export-swap.ts` with a single pair:

```ts
export function swapExportsForPublish(pkgJsonPath: string): Restore;
export function restoreExports(pkgJsonPath: string, restore: Restore): void;
```

Have `publish.ts`, `prepack-exports.ts`, and (if kept) `release.ts` all import this. Expected savings: ~80 lines across three files, plus eliminated drift between the three backup-file conventions.

## 4. 🟠 CLI imports mix subpath exports and relative paths

`packages/mcp/package.json:21-84` exposes **12 subpath exports** (`@dryui/mcp/reviewer`, `/theme-checker`, `/project-planner`, `/workspace-audit`, `/composition-search`, `/spec-formatters`, `/toon`, `/tokens`, `/composition-data`, `/spec-types`, `/architecture`, `/spec`). Half of them are consumed correctly by the CLI; the other half are imported via `../../../mcp/src/*.js` relative paths.

### Evidence

`rg "from ['\"]\\.\\./\\.\\./\\.\\./mcp" packages/cli`

```
commands/review.ts:4:      import { reviewComponent }     from '../../../mcp/src/reviewer.js'
commands/diagnose.ts:4:    import { diagnoseTheme }       from '../../../mcp/src/theme-checker.js'
commands/detect.ts:3:      import { detectProject }       from '../../../mcp/src/project-planner.js'
commands/install.ts:3:     import { planInstall }         from '../../../mcp/src/project-planner.js'
commands/add.ts:6:         import { planAdd }             from '../../../mcp/src/project-planner.js'
commands/init.ts:12:                                      from '../../../mcp/src/project-planner.js'
commands/project-planner.ts:12:                            from '../../../mcp/src/project-planner.js'
commands/doctor.ts:3:      import { buildWorkspaceReport } from '../../../mcp/src/workspace-audit.js'
commands/lint.ts:3:        import { buildWorkspaceReport } from '../../../mcp/src/workspace-audit.js'
commands/workspace-args.ts:3: import type { WorkspaceSeverity } from '../../../mcp/src/workspace-audit.js'
```

Meanwhile, `info.ts`, `list.ts`, `tokens.ts`, `compose.ts`, `run.ts`, `index.ts`, `format.ts` all correctly use `@dryui/mcp/<subpath>`. The split is arbitrary — both halves target the same `src/*.ts` files via the `"bun": "./src/*.ts"` condition in the exports map.

### Impact

- A consumer who installs `@dryui/cli` standalone gets broken builds because `../../../mcp/src/*.js` resolves to nothing outside the monorepo.
- The CLI tests pass only because the monorepo layout accidentally satisfies the relative path. This is the kind of bug that surfaces only when you try to publish.
- Renaming a file in `packages/mcp/src/` will only fail half the CLI — the half that uses subpath exports keeps working because the exports map is the contract, while the relative-import half silently breaks.

### Fix

Mechanical edit — replace in each file:

```diff
-from '../../../mcp/src/reviewer.js'
+from '@dryui/mcp/reviewer'
```

Add a lint rule: `packages/cli/src/**` may not import `../../mcp` or `../../../mcp`. Enforce via an ESLint `no-restricted-imports` rule with a regex, or a small grep-based check added to `validate.ts`.

### Aside: `scanWorkspace` vs `buildWorkspaceReport`

`packages/mcp/src/workspace-audit.ts` exports **two names for the same function** — `scanWorkspace()` (used by the MCP server) and `buildWorkspaceReport()` (used by the CLI). Pick one, alias or re-export the other for one release cycle, then remove the alias. No behavioral change, just nomenclature hygiene.

## 5. 🟠 CSS patterns that should be tokens

The `@dryui/ui` package's scoped styles have a handful of patterns so widely repeated that they're de facto unwritten design tokens. Counts below are from direct `rg` runs, not the agent estimates.

### Top offenders

| Pattern                                                                                | Verified count                         | Top files                                                                                                                                                                                                              |
| -------------------------------------------------------------------------------------- | -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `outline: 2px solid var(--dry-color-focus-ring)` (outset focus ring)                   | **37** occurrences across **34** files | `button`, `input`, `textarea`, `tags-input`, `file-select`, `drag-and-drop-handle`, `phone-input-select`, `table-of-contents-item`, `slider-input`, `combobox-input`, `color-picker-*`, `tree-item-label`, `card-root` |
| `outline: 2px solid var(--dry-color-focus-ring); outline-offset: -1px` (inset variant) | 17 occurrences across 16 files         | `input`, `textarea`, `combobox-input`, `color-picker-input`, `file-select-root`, `phone-input-select`, `label`, `number-input-button`, `pin-input-root`, `slider-input`                                                |
| `opacity: 0.5; cursor: not-allowed` (disabled state)                                   | 17 occurrences across 16 files         | `textarea`, `file-upload-dropzone`, `command-palette-item`, `pin-input-root`, `slider-input`, `color-picker-*`, `list-item`, `file-select-root`, `phone-input-select`, `label`, `number-input-button`                  |
| `transition: ... \d+m?s ...` (raw duration literal — should use `--dry-duration-*`)    | 13 occurrences across 11 files         | `flip-card-*`, `tree-item-children`, `video-embed-button`, `option-picker-preview`, `progress`, `gauge`, `chromatic-shift`, `code-block-button`, `notification-center-item`, `toast-root`                              |
| `appearance: none`                                                                     | 30 occurrences                         | all form controls                                                                                                                                                                                                      |

Beyond these top five, the component-local private-token proliferation is worth flagging: `button.svelte` alone defines 18+ `--_dry-btn-*` private variables for soft/solid/ghost/outline variants. `input.svelte`, `textarea.svelte`, `combobox-input.svelte`, `select-trigger.svelte`, and `color-picker-input.svelte` each define 4-5 nearly-identical `--_dry-*-bg`, `--_dry-*-border`, `--_dry-*-color`, `--_dry-*-radius` tokens independently.

### Impact

- Changing the focus-ring thickness from 2px to 1.5px is a **34-file PR**, not a one-line token change.
- The "disabled state" concept is expressed in 16 places with no guarantee they agree (I verified they don't — some use `0.5`, one uses `0.6`).
- A designer asking "where do I change how all inputs look" gets no single answer.

### Fix — priority order

1. **Add three semantic shorthands** to `packages/ui/src/themes/default.css`:
   - `--dry-focus-ring: 2px solid var(--dry-color-focus-ring)` → so components write `outline: var(--dry-focus-ring); outline-offset: 2px;`
   - `--dry-state-disabled-opacity: 0.5` → replaces every raw `0.5`
   - `--dry-duration-fast: 150ms; --dry-duration-med: 250ms; --dry-ease: cubic-bezier(...)` (already exist — just **enforce** use in the 13 holdouts with a lint rule)
2. **Unify form-control tokens.** Add a `--dry-form-control-*` family (bg, bg-hover, bg-disabled, border, border-hover, border-focus, radius, font-size, padding-block, padding-inline). Let `input`, `textarea`, `combobox-input`, `color-picker-input`, `select-trigger`, `phone-input-select`, `file-select-root`, `pin-input-cell` all consume from it.
3. **Don't** try to unify the button soft variant color-mixes yet — they are genuinely component-specific. Revisit if a second component needs "soft color scales" (Tag? Badge? Chip?), at which point promote to `--dry-soft-*` family.

Lint rule to add in `@dryui/lint`: `dryui/prefer-focus-ring-token`. It's a one-line regex check and would keep the new token honest.

## 6. 🟠 Skill sync leaves `.codex` drifting

`scripts/sync-skills.ts` is supposed to one-way copy `packages/ui/skills/dryui/` → `.claude/skills/dryui/` → `.codex/skills/dryui/`. Verified state:

```
$ diff .claude/skills/dryui/SKILL.md packages/ui/skills/dryui/SKILL.md
(no output — identical)

$ diff .codex/skills/dryui/SKILL.md packages/ui/skills/dryui/SKILL.md
Files .codex/skills/dryui/SKILL.md and packages/ui/skills/dryui/SKILL.md differ
```

The `.codex` copy still references `OptionSwatchGroup` in its compound-components list; the source and the `.claude` copy have `OptionPicker`. This matches `git status`, which shows `apps/docs/src/lib/demos/OptionSwatchGroupDemo.svelte` deleted and the SKILL.md modified. Someone ran sync against `.claude` but not `.codex`, or `sync-skills.ts` doesn't iterate all targets on every run.

### Fix

Open `scripts/sync-skills.ts` and confirm it writes to **every** target on every run — not just the ones whose destination already differs from source. This is a one-function audit. Add a test that asserts post-sync, `diff .codex/skills/dryui/SKILL.md packages/ui/skills/dryui/SKILL.md` is empty.

A better long-term fix: **use a symlink or a copy-on-build step.** The three-way copy is inherently drift-prone. If Claude Code and Codex don't follow symlinks, then make the source-of-truth copy live at `.claude/skills/dryui/` (the one copy that's already in sync) and have `packages/ui/skills/dryui/` be a build artifact that `scripts/sync-skills.ts` _generates_, not reads from.

## 7. 🟠 Browser test boilerplate

Every file under `tests/browser/*.browser.test.ts` — **28 files** — opens with the same 10-line block:

```ts
const mountedComponents: ReturnType<typeof mount>[] = [];
afterEach(() => {
	for (const component of mountedComponents.splice(0)) {
		unmount(component);
	}
	document.body.replaceChildren();
});
```

And then every file defines its own `renderComponentName(props)` factory inline — 28 near-identical helpers, differing only by which import they call `mount()` on.

### Fix

One file: `tests/browser/_harness.ts`.

```ts
const mounted: Mounted[] = [];
afterEach(() => {
	for (const c of mounted.splice(0)) unmount(c);
	document.body.replaceChildren();
});

export function render<T>(
	Component: ComponentConstructor<T>,
	props: T = {} as T,
	target: HTMLElement = document.body
) {
	const instance = mount(Component, { target, props });
	mounted.push(instance);
	return { instance, target };
}
```

Each test becomes `import { render } from './_harness'` + `render(Button, { variant: 'solid' })`. Saves ~280 lines and removes the drift risk of 28 subtly-different `render` helpers.

**Don't touch the split** between `tests/unit/` (Bun + happy-dom) and `tests/browser/` (Vitest + Playwright) — that split is principled. Different runners, different guarantees, both needed.

## 9. 🟡 GitHub workflows duplicate setup

`.github/workflows/release.yml` and `.github/workflows/ci.yml` (which is actually a docs-deploy workflow despite the name — rename it to `deploy-docs.yml`) share the same 5-step preamble:

```yaml
- actions/checkout@v5
- oven-sh/setup-bun@v2 with bun-version: 1.3.10
- actions/cache@v5 (Bun cache)
- run: bun install --frozen-lockfile
- run: bun run build:packages # or the two-filter equivalent
```

Both workflows run on every push to `main`. That's ~20 lines duplicated and two machines doing the identical install + build.

### Fix

1. **Rename `.github/workflows/ci.yml` → `deploy-docs.yml`.** The current name is misleading — per `CLAUDE.md`, there is no actual CI/PR validation workflow.
2. Extract `.github/actions/setup-dryui/action.yml` (composite action) with the checkout/setup-bun/cache/install steps. Both workflows become:

   ```yaml
   - uses: ./.github/actions/setup-dryui
   - run: bun run build:packages
   ```

3. **Add a PR validation workflow.** `CLAUDE.md` explicitly notes: "No PR validation workflow — run `bun run validate` locally before pushing." That's a gap, not a feature. `validate.ts` is the _only_ gate on the codebase, and it only runs when a human remembers to run it. Create `.github/workflows/validate.yml` that runs on `pull_request` and calls `bun run validate --no-test` (or the full thing if runtime permits).

This last point is arguably severity 🟠, but I'm leaving it as 🟡 because it's adjacent to duplication rather than caused by it.

## 10. 🟡 The export-syncing triple

Three scripts touch package export maps:

| Script                                        | When it runs                                               | What it does                                                                                                              |
| --------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `scripts/sync-package-exports.ts` (162 lines) | `bun run check:exports`, called from `validate.ts`         | Generates `exports` (pointing at `src/`) and `publishConfig.exports` (pointing at `dist/`) from `index.ts` barrel imports |
| `scripts/prepack-exports.ts` (26 lines)       | `npm pack` prepack hook (packages/ui, packages/primitives) | Swaps `publishConfig.exports` → `exports`, backs up original                                                              |
| `scripts/postpack-exports.ts` (23 lines)      | `npm pack` postpack hook                                   | Restores from backup                                                                                                      |

Individually each has a single job. Collectively they form a fragile three-stage lifecycle with no test that asserts `sync` → `prepack` → `postpack` is a round-trip. If `sync-package-exports.ts` generates a new publishConfig entry but prepack doesn't see it because the backup file is from a previous run, the published tarball will contain wrong paths.

### Fix

1. Unify naming. Rename `sync-package-exports.ts` → `generate-exports.ts`. Rename `prepack-exports.ts` / `postpack-exports.ts` → `apply-publish-exports.ts` / `restore-publish-exports.ts`.
2. Extract the shared swap helper per §3.
3. Add a smoke test: run generate → apply → restore in a temp dir and assert file-state equality before and after.
4. Consider replacing the three scripts with a single `scripts/exports.ts <generate|apply|restore>` subcommand dispatcher — same logic, one entrypoint.

## 11. 🟡 tsconfig.json near-duplicates

`packages/primitives/tsconfig.json`, `packages/ui/tsconfig.json`, `packages/theme-wizard/tsconfig.json`, `packages/feedback/tsconfig.json` are 3-4 line files all extending `../../tsconfig.json` with identical `include` and `paths`. `packages/lint/tsconfig.json` and `packages/feedback-server/tsconfig.json` form a second 5-line identical group for Node packages.

### Fix

Create two templates in the repo root:

- `tsconfig.svelte-package.json` — extends `tsconfig.json`, Svelte-aware defaults, `include` defaults for `src/`
- `tsconfig.node.json` — extends `tsconfig.json`, Node/ESM defaults

Per-package files become one line:

```json
{ "extends": "../../tsconfig.svelte-package.json" }
```

Only `packages/cli` and `packages/mcp` (the two with meaningful overrides) keep custom bodies.

## 12. 🔵 Findings that turned out not to be duplication

I want to note these explicitly so we don't re-litigate them later:

- **Tabs / SegmentedControl `ui/` vs `primitives/`** — NOT duplicates. `packages/ui/src/tabs/tabs-root.svelte` and `packages/primitives/src/tabs/tabs-root.svelte` differ in exactly the ways they should: the UI version adds `class={className}`, `data-tabs-root`, and a `<style>` block that sets component tokens. Same for `segmented-control-root.svelte`. An earlier sweep flagged these as copies; the diff (verified) shows they are correct extensions. Leave them alone.

- **`@dryui/plugin` as a "dead package"** — NOT dead. It contains `.claude-plugin/`, `.codex-plugin/`, `.mcp.json`, `assets/`, and `skills/`, and exists to be the plugin bundle for Claude Code / Codex marketplaces. It has no source because it _is_ the manifest. Keep it, but document its purpose in a one-line comment in `package.json` so the next auditor doesn't flag it.

- **MCP server `index.ts` at 778 lines** — looks monolithic but is actually well-structured: a linear wiring file that declares each of the 12 tools in turn, importing implementations from sibling modules (`reviewer.ts`, `theme-checker.ts`, `project-planner.ts`, `workspace-audit.ts`, `composition-search.ts`, `toon.ts`). It is not duplicating logic; it is the MCP-protocol boilerplate. Don't refactor for the sake of LOC.

- **FileSelect tested twice** — once as a Bun unit test (`tests/unit/file-select.test.ts`) and once as a Vitest browser test (`tests/browser/file-select.browser.test.ts`). Different layers: the unit test verifies exports and shape; the browser test verifies the `<input type="file">` interaction. Keep both.

- **UI context `.svelte.ts` files vs primitives context files** — all 54 UI context files `import { createContext } from '@dryui/primitives'` (verified). They are not reimplementing the primitive; they are type-scoped instances of it. Correct usage.

- **llms.txt variants (`/llms.txt`, `apps/docs/static/llms.txt`, `apps/docs/static/llms-components.txt`, `apps/docs/static/get-started.txt`)** — different sizes (584 / 1354 / 4151 / 231 lines), different purposes (root summary / detailed reference / full spec / install playbook). Intentionally tiered. Document the tiering in a comment at the top of each file so the next auditor doesn't think they're copies; no consolidation needed.
