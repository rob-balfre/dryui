# DryUI DRY Principle Audit

Date: 2026-04-13

Scope: repository-wide audit of author-maintained source, scripts, docs, skills, plugin metadata, and workspace tooling.

Excluded from duplication counts: `node_modules/`, `dist/`, `.svelte-kit/`, `vendor/`, `tmp/`, `build/`, lockfiles, and binary assets.

## Executive Summary

DryUI already has a few strong single-source-of-truth patterns:

- package export maps are generated by [`scripts/sync-package-exports.ts`](../scripts/sync-package-exports.ts)
- composition guidance is centralized in [`packages/mcp/src/composition-data.ts`](../packages/mcp/src/composition-data.ts)
- the docs tool/CLI capability cards already reuse [`packages/mcp/src/ai-surface.ts`](../packages/mcp/src/ai-surface.ts)

The main DRY failures are not in those areas. They are concentrated in five places:

1. skill content and agent setup instructions exist in multiple living copies
2. release and publish logic has more than one authoritative path
3. validation still parses prose and hand-maintained lists instead of structured metadata
4. the repo’s own audit tooling is out of sync with official theme files
5. the component surface contains many exact-copy shell components and thin wrappers that are still hand-maintained

## Methodology

This audit combined four passes:

1. structural review of root scripts, package manifests, workflows, plugin metadata, docs, and skills
2. exact duplicate-file hashing on non-generated text sources
3. workspace-level DryUI audit via `doctor` / `lint`
4. manual inspection of the highest-signal hotspots before making any recommendation

## What Is Already Healthy

### Export map generation is already centralized

[`scripts/sync-package-exports.ts`](../scripts/sync-package-exports.ts) is a good pattern. It derives the `exports` and `publishConfig.exports` maps for `@dryui/primitives` and `@dryui/ui` from the public barrels and keeps them synchronized with `--check`.

Recommendation: keep this pattern and copy it anywhere the repo still depends on hand-maintained inventories.

### Composition guidance already has a real canonical source

[`packages/mcp/src/composition-data.ts`](../packages/mcp/src/composition-data.ts) is explicitly called out in [`CLAUDE.md`](../CLAUDE.md) as the canonical source for composition rules. Both MCP and CLI consume it.

Recommendation: use this as the model for skills, docs navigation metadata, and agent install/setup content.

### Tool and CLI capability cards are partially centralized

[`apps/docs/src/lib/ai-setup.ts`](../apps/docs/src/lib/ai-setup.ts) already imports [`packages/mcp/src/ai-surface.ts`](../packages/mcp/src/ai-surface.ts) for tool and CLI descriptions.

Recommendation: extend that pattern so install/setup snippets also come from a single manifest instead of being repeated in docs and markdown prose.

## Findings

## P0. Skills And Setup Instructions Have Multiple Living Copies

### Why this matters

This is the clearest single-source-of-truth failure in the repo. The same DryUI skill is authored once, copied into multiple tool-specific locations, and then documented again in multiple markdown files. That is already drifting.

### Evidence

- Canonical-looking source skill: [`packages/ui/skills/dryui/SKILL.md`](../packages/ui/skills/dryui/SKILL.md)
- Tracked duplicate copy: [`.claude/skills/dryui/SKILL.md`](../.claude/skills/dryui/SKILL.md)
- Local generated copy: [`.codex/skills/dryui/SKILL.md`](../.codex/skills/dryui/SKILL.md)
- Replication mechanism: [`scripts/sync-skills.ts`](../scripts/sync-skills.ts) lines 146-177
- Current drift:
  - [`packages/ui/skills/dryui/SKILL.md`](../packages/ui/skills/dryui/SKILL.md) lists `OptionPicker`
  - [`.claude/skills/dryui/SKILL.md`](../.claude/skills/dryui/SKILL.md) also lists `OptionPicker`
  - [`.codex/skills/dryui/SKILL.md`](../.codex/skills/dryui/SKILL.md) still lists `OptionSwatchGroup`
- Repeated AI setup instructions exist in four places:
  - [`README.md`](../README.md) lines 92-214
  - [`AGENTS.md`](../AGENTS.md) lines 10-41
  - [`CLAUDE.md`](../CLAUDE.md) lines 96-136
  - [`apps/docs/src/lib/ai-setup.ts`](../apps/docs/src/lib/ai-setup.ts) lines 92-249

### Concrete drift already visible

- `OptionSwatchGroup` is still named in [`README.md`](../README.md) line 86
- the live code exports `OptionPicker` from [`packages/ui/src/index.ts`](../packages/ui/src/index.ts)
- the docs nav already uses `OptionPicker` in [`apps/docs/src/lib/nav.ts`](../apps/docs/src/lib/nav.ts)

This is exactly the kind of drift a DRY audit is supposed to catch: the product changed once, but the same fact still exists in several manual copies.

### Recommendation

Choose one real source for skill and setup content:

| Domain                      | Canonical source                                     | Generated consumers                               |
| --------------------------- | ---------------------------------------------------- | ------------------------------------------------- |
| DryUI skill content         | `packages/ui/skills/dryui/`                          | `.claude/`, `.codex/`, `.cursor/rules/`           |
| Tool install/setup snippets | new `packages/mcp/src/install-surface.ts` or similar | `README.md`, `AGENTS.md`, `CLAUDE.md`, docs pages |

Then make the repo policy explicit:

1. only `packages/ui/skills/dryui/` is committed as source
2. `.claude/skills/**` becomes generated and should not be hand-edited
3. `.codex/skills/**` and `.cursor/rules/**` remain generated only
4. `README.md`, `AGENTS.md`, and `CLAUDE.md` should consume shared generated fragments or include only links to the canonical docs page

### Priority

Highest. This is already drifting, and the repo explicitly markets its skill/plugin/MCP story as a product surface.

## P0. Release And Publish Have More Than One Authoritative Path

### Why this matters

The repo currently has multiple release stories, but they are not all active and they do not point to the same implementation.

### Evidence

- Active root scripts in [`package.json`](../package.json):
  - `publish:packages = "changeset publish"` at line 43
  - `release = "bun run validate --no-test && bun run version && changeset publish"` at line 44
- Unused custom release path: [`scripts/release.ts`](../scripts/release.ts)
- Unused single-package publish path: [`scripts/publish.ts`](../scripts/publish.ts)
- `scripts/release.ts` duplicates logic also present elsewhere:
  - `publishConfig` swap
  - workspace dependency rewriting
  - restore backups
- `scripts/publish.ts` duplicates the same responsibilities for a single package
- `scripts/prepack-exports.ts` and [`scripts/postpack-exports.ts`](../scripts/postpack-exports.ts) already handle export rewriting for `@dryui/primitives` and `@dryui/ui`
- I found no `workspace:*` dependencies in published package manifests, so the workspace-version rewrite logic in `scripts/release.ts` and `scripts/publish.ts` is dead weight right now

### Structural problem

There are effectively four release narratives:

1. root `package.json` scripts
2. `scripts/release.ts`
3. `scripts/publish.ts`
4. `.github/workflows/release.yml`

That is too many for a repo this size, especially for a high-risk path like publishing.

### Recommendation

Pick one path and delete the others.

Preferred approach:

1. keep `changesets/action` in GitHub Actions
2. keep one local manual release command
3. if custom publish handling is still necessary, make the root `release` script call the custom implementation directly
4. otherwise delete `scripts/release.ts` and `scripts/publish.ts`
5. move release guidance in markdown files to a single short pointer to the canonical command

### Priority

Highest. This is an operational single-source-of-truth problem.

## P0. Validation Still Parses Prose And Hand-Maintained Lists Instead Of Structured Metadata

### Why this matters

The repo already has metadata, but some important checks still depend on English sentences and manual nav wiring.

### Evidence

- [`scripts/validate-spec-coverage.ts`](../scripts/validate-spec-coverage.ts) parses:
  - `COMPONENT_META` from source text in `generate-spec.ts` lines 67-108
  - the docs nav from regex against `apps/docs/src/lib/nav.ts` lines 138-146
  - the compound-component list by regex against the exact sentence in `packages/ui/skills/dryui/SKILL.md` lines 162-177
- the script explicitly fails if the phrase `"Compound components include ..."` changes
- the script also maintains manual stale-pattern guards for removed layout primitives at lines 32-53 and 330-340

### Why this is brittle

The repo has the right intent, but the contract is split across:

- source directories
- `COMPONENT_META`
- public barrels
- `spec.json`
- docs nav
- `SKILL.md` prose
- additional stale-string allow/deny lists

That is not one source of truth. It is a synchronization ring.

### Recommendation

Introduce one structured component manifest with fields like:

| Field        | Purpose                                    |
| ------------ | ------------------------------------------ |
| `name`       | public component name                      |
| `dir`        | source directory                           |
| `public`     | exported or latent                         |
| `compound`   | structured compound status                 |
| `docs`       | docs-nav category / visibility             |
| `skill`      | whether it should appear in skill guidance |
| `deprecated` | old names / migration aliases              |

Then:

1. generate `spec.json` from it
2. generate the compound-component skill sentence from it
3. generate the docs nav leaf inventory from it
4. generate architecture mismatch checks from it

The current validation script is useful, but it is enforcing consensus across multiple manual copies rather than enforcing a true canonical source.

### Priority

Highest. This is a root-cause issue behind a lot of the visible drift.

## P1. The Repo’s Own Theme Checker And Official Themes Are Out Of Sync

### Why this matters

A repo-level audit tool that flags the official theme files creates noise and reduces trust in the checker. That is a DRY failure because the token contract exists in more than one place.

### Evidence

Workspace audit summary:

- 551 findings total
- top rule: `theme/unknown-component-token` with 208 occurrences

Top files by finding count:

- `packages/ui/src/themes/default.css`: 91
- `packages/ui/src/themes/midnight.css`: 54
- `packages/ui/src/themes/terminal.css`: 53
- `packages/ui/src/themes/aurora.css`: 41
- `packages/ui/src/themes/dark.css`: 30

Concrete examples from official themes:

- `--dry-toggle-track-bg`
- `--dry-toggle-thumb-bg`
- `--dry-type-display-size`
- `--dry-type-ui-title-leading`

The checker says these are not recognized component tokens even though they live in the first-party theme files.

### Interpretation

Either:

1. the official themes are defining tokens that are not represented in the spec, or
2. the checker/spec token inventory is incomplete

In both cases the same contract is split.

### Recommendation

Create one token registry that both theme files and audit tooling consume.

Possible canonical source:

- generated manifest from component CSS variable declarations plus theme semantic tokens
- or a checked-in token manifest consumed by both the theme checker and spec generator

Then:

1. make `theme-checker` read that registry
2. make official theme files validate cleanly
3. fail CI if first-party theme files produce unknown-token warnings

### Priority

High. This is a trust issue in the repo’s own tooling.

#### Additional exact-copy groups

### Recommendation

Do not collapse the public API names. Instead, stop hand-authoring the zero-logic shells.

Preferred options:

1. generate passthrough primitive shells from a tiny template
2. introduce a small internal factory for the no-logic parts
3. keep the files only where a part carries real semantics, behavior, or style

Also review the dead or latent layout files:

- `packages/primitives/src/flex/`
- `packages/primitives/src/grid/`
- `packages/primitives/src/stack/`

They are no longer publicly exported, but they still exist in source and show up in duplication scans.

### Priority

High, but this should be handled with codegen/templates, not an indiscriminate “merge components” refactor.

## P1. CLI Command Modules And Their Tests Repeat The Same Skeleton

### Why this matters

The CLI is thin by design, but it still has a lot of boilerplate repeated across commands and parallel test fixtures that largely mirror MCP tests.

### Evidence

Command boilerplate is repeated across:

- `packages/cli/src/commands/detect.ts`
- `packages/cli/src/commands/install.ts`
- `packages/cli/src/commands/review.ts`
- `packages/cli/src/commands/diagnose.ts`
- `packages/cli/src/commands/doctor.ts`
- `packages/cli/src/commands/lint.ts`
- `packages/cli/src/commands/list.ts`
- `packages/cli/src/commands/info.ts`
- `packages/cli/src/commands/tokens.ts`
- `packages/cli/src/commands/compose.ts`

Common repeated elements:

- `--help` handling
- `resolveOutputMode(...)`
- `switch (mode)` with TOON / JSON / text branches
- `runCommand(...)`

The test layer also mirrors MCP tests instead of sharing fixtures:

- `packages/mcp/src/project-planner.test.ts`
- `packages/cli/src/__tests__/project-planner.test.ts`
- `packages/mcp/src/workspace-audit.test.ts`
- `packages/cli/src/__tests__/workspace-audit.test.ts`

Both CLI test files reintroduce temporary workspace builders and `captureCommand(...)` helpers instead of reusing a shared harness.

### Recommendation

Introduce a tiny internal command factory plus shared test helpers:

| Area                            | Proposed extraction                                                   |
| ------------------------------- | --------------------------------------------------------------------- |
| command help/output boilerplate | `defineCommand(...)` helper                                           |
| file-based commands             | `defineFileCommand(...)` helper                                       |
| workspace commands              | shared factory for `doctor` and `lint`                                |
| CLI command tests               | `createTempWorkspace(...)`, `captureCommand(...)`, `mockSpec` helpers |

This will not transform the product, but it will make the CLI easier to extend without repeating the same plumbing.

### Priority

High-medium.

## P2. Documentation Drift Exists Because Markdown Is Repeating Operational Facts

### Why this matters

The repo contains multiple long-form operational documents. That is fine until they each restate commands and environment facts in full.

### Evidence

Current drift examples:

- [`README.md`](../README.md) line 86 still lists `Option Swatch Group`
- the live product uses `OptionPicker`
- [`README.md`](../README.md) lines 233-234 tell contributors to run `apps/playground`, which no longer exists
- [`README.md`](../README.md) lines 278-282 say `bun run publish`, but there is no `publish` script in [`package.json`](../package.json)
- npm token rotation guidance is repeated in:
  - `README.md`
  - `AGENTS.md`
  - `CLAUDE.md`
- AI setup/install instructions are repeated in:
  - `README.md`
  - `AGENTS.md`
  - `CLAUDE.md`
  - `apps/docs/src/lib/ai-setup.ts`

### Recommendation

Split docs into:

1. one canonical operational reference page
2. one canonical agent-setup manifest
3. short pointers from `README.md`, `AGENTS.md`, and `CLAUDE.md`

The markdown files should link to canonical material, not restate it all.

### Priority

Medium.

## P2. Existing Architecture Audit Infrastructure Is Underused

### Why this matters

The repo already has an architecture audit report and generator, but it is not yet capturing the real duplication hotspots.

### Evidence

[`reports/architecture-audit.md`](../reports/architecture-audit.md) reports:

- `Audit clusters: 0`
- `No findings in this bucket` for all duplication-priority sections

Yet the same report also reports:

- 64 thin wrapper candidates
- 15 primitive exports missing spec metadata

The plumbing is there, but the duplication-cluster layer is empty.

### Recommendation

Extend the architecture audit generator so CI can report:

- exact duplicate shell components
- identical wrapper entrypoints
- latent non-exported components
- skill/docs/install drift against the canonical manifest
- theme-token registry drift

The repo already has the right reporting surface. It just needs more real signals wired into it.

### Priority

Medium.

- `apps/docs/src/routes/theme-wizard/typography/+page.server.ts`

If these routes remain, centralize the redirect helper or use a more compact route strategy.

### Shared tsconfig files

Identical config pairs:

- `packages/cli/tsconfig.json`
- `packages/mcp/tsconfig.json`

and:

- `packages/feedback-server/tsconfig.build.json`
- `packages/mcp/tsconfig.build.json`

### `.DS_Store` noise

## Proposed Single Source Of Truth Map

| Domain                         | Current state                    | Proposed canonical source                            |
| ------------------------------ | -------------------------------- | ---------------------------------------------------- |
| component export maps          | mostly centralized               | keep `scripts/sync-package-exports.ts`               |
| composition guidance           | centralized                      | keep `packages/mcp/src/composition-data.ts`          |
| tool + CLI descriptions        | centralized                      | keep `packages/mcp/src/ai-surface.ts`                |
| skill content                  | duplicated                       | `packages/ui/skills/dryui/` only                     |
| agent install/setup snippets   | duplicated                       | new generated `install-surface` manifest             |
| release flow                   | split                            | root release command + release workflow only         |
| publish internals              | split                            | either prepack hooks or one release script, not both |
| component/docs/skill inventory | synchronization ring             | one structured component manifest                    |
| theme token registry           | split between themes and checker | one generated token manifest                         |
| passthrough shell components   | hand-authored duplicates         | template or codegen                                  |
| CLI command plumbing           | repeated                         | command factory + shared test helpers                |

## Recommended Order Of Operations

1. Fix the skill/setup/documentation surface first.
2. Collapse the release/publish flow to one path.
3. Replace prose parsing with structured component metadata.
4. Make official theme files validate cleanly against the checker.
5. Introduce templates/codegen for exact-copy primitive shells.
6. Refactor CLI command/test boilerplate only after the product metadata is stabilized.

## Suggested First PRs

### PR 1. Canonicalize Skill And Setup Content

- keep only `packages/ui/skills/dryui/` as source
- stop tracking `.claude/skills/dryui/`
- introduce shared install/setup data for docs and markdown docs
- update stale `OptionSwatchGroup` references

- delete `scripts/release.ts` and `scripts/publish.ts` if they are no longer needed
- or wire root `release` directly through the one script that matters
- update docs so every release instruction points at one command

### PR 3. Replace Prose Parsing With Structured Metadata

- add one component manifest
- generate:
  - compound component list in the skill
  - docs nav leaf inventory
  - spec coverage checks

### PR 4. Fix Theme Token Contract Drift

- generate or define the token registry
- make `theme-checker` trust the same registry official themes use

### PR 5. Template The Zero-Logic Shell Components

- start with the 20 exact-copy primitive passthrough files
- do not change the public API
- remove hand-maintained duplication, not the semantics

## Bottom Line

DryUI’s biggest DRY problem is not “too many components.”

It is that the repo has several places where product truth is duplicated across:

- markdown prose
- skill copies
- docs setup code
- release scripts
- validation heuristics

The fastest way to improve the project is to move those domains onto generated or structured manifests first. After that, the smaller code-level duplications become much easier to simplify safely.
